                                                                                                                                  # 知识地图（网络知识图）生成方案 Spec

## Why
当文章数量增多时，单纯的关键词搜索难以支持“主题回顾、脉络梳理、关联阅读”。需要用自动化（可含 AI agent）把文章组织成可浏览的知识地图，以提升内容消费与内容优化效率。

## What Changes
- 增加“知识地图”能力：从文章集合中生成多个“图”视角（主题/概念/引用/相似度/时间演化等），并提供可视化浏览入口
- 明确至少 10 种**思考模式不同**的生成方案（不是换工具套壳），每种方案给出：输入信号、生成逻辑、产出物、资源/运维画像、适用场景与风险
- 给出默认推荐的最小实现路径（省心、资源友好、自建优先），并说明何时升级到更复杂方案

## Impact
- Affected specs: 内容组织与发现、信息架构、搜索与推荐、数据分析闭环
- Affected code: 文章索引/元数据、构建流水线、前端可视化页面、可选的轻量后端/数据库

## ADDED Requirements

### Requirement: 知识地图入口
系统 SHALL 提供“知识地图”入口页面，展示至少一种可交互的地图视图，支持从节点跳转到文章详情页。

#### Scenario: 浏览关联阅读
- **WHEN** 用户打开知识地图并点击某个节点
- **THEN** 页面展示该节点的关联节点摘要，并可跳转到对应文章

### Requirement: 多视角地图
系统 SHALL 支持至少 3 种不同“信号来源/建图逻辑”的地图视角（例如：标签共现、相似度、引用/内链）。

### Requirement: 可离线生成与低资源运行
系统 SHALL 支持在构建阶段（本地/CI）离线生成地图产物；线上运行期 SHOULD 仅托管静态产物或极轻量服务，以适配 2C/2G/3Mbps/40G 环境。

### Requirement: 可增量更新
系统 SHALL 支持增量更新：新增/修改文章时，只重新计算受影响部分（例如新文章的边、或其相似度邻居），避免全量重算成为负担。

### Requirement: 数据可导出
系统 SHALL 以可移植格式输出地图产物（JSON/CSV/GraphML 等），支持后续更换前端可视化实现与迁移。

### Requirement: 隐私与安全
系统 SHALL 默认不把完整文章内容发送到第三方服务；如用户显式启用外部模型/接口，必须提供可切换与脱敏策略（例如只发送摘要/标题/关键词）。

## 10+ 种方案（思考模式不同，非换工具套壳）

下面每种方案都用不同“信号/建模假设”组织文章，适用于不同目标。可以并存：同一站点提供多张地图（Tab 切换）。

### 方案 1：人工结构优先（目录/系列/手工边）+ 自动补全
**核心思路**：把作者的“写作意图结构”作为第一原则，AI 只做补全而不是主导。
- 输入信号：文章 frontmatter（series、order、parent、related）、手工维护的 links 文件
- 建图逻辑：显式边为主；缺失边由规则补齐（同系列、同标签、同目录相邻）
- 产出物：有向图（系列/章节）+ 关联阅读边（弱）
- 资源画像：极低；无模型依赖；最可控
- 风险/代价：需要少量维护纪律；但最“省心稳定”

### 方案 2：内链/引用图（WikiLink/脚注/引用）驱动
**核心思路**：把文章之间的“显式引用关系”视作知识脉络，强调可解释性。
- 输入信号：文章中的链接（站内链接、引用区、脚注），可扩展解析 Markdown
- 建图逻辑：解析站内 URL 形成有向边；边权可按链接位置（正文/脚注）加权
- 产出物：引用网络 + 反向链接列表（Backlinks）
- 资源画像：低；完全离线可做
- 风险/代价：对“有没有写内链”敏感；需要写作时习惯加链接

### 方案 3：标签/关键词共现图（统计学共现）
**核心思路**：把标签（或抽取关键词）当作“概念”，用共现形成概念网络与文章聚类。
- 输入信号：标签、标题分词、摘要关键词（可规则/轻量抽取）
- 建图逻辑：共现矩阵 -> 阈值化或 PMI/TF-IDF 加权 -> 生成无向图
- 产出物：概念图（tag graph）+ 文章到概念的二部图
- 资源画像：低；无需大模型；易增量更新
- 风险/代价：标签质量决定上限；需要治理同义词（可加 alias）

### 方案 4：相似度近邻图（Embedding kNN）
**核心思路**：把每篇文章编码成向量，按相似度连边，得到“语义邻居网络”。
- 输入信号：标题+摘要（优先）或正文；embedding 模型可本地或外部
- 建图逻辑：向量化 -> kNN（每篇连 k 个最相似文章）-> 边权为相似度
- 产出物：文章-文章相似图 + “推荐阅读”列表
- 资源画像：中（取决于模型与文章数）；可离线；线上只读静态 JSON
- 风险/代价：可解释性弱；需要控制“相似但不相关”的误连（阈值/多信号校验）

### 方案 5：实体-关系抽取图（Knowledge Graph）
**核心思路**：从文本抽取实体与关系，形成“概念/人物/技术”级别的知识图谱。
- 输入信号：正文/摘要；实体词表（可选）与抽取器（规则/模型/LLM）
- 建图逻辑：NER + relation extraction -> 归一化（同义合并）-> 实体图 + 文章证据
- 产出物：实体图（entity graph）+ 每条边的证据链接（指向文章段落）
- 资源画像：中到高；需要较多质量控制；但最“像知识图”
- 风险/代价：抽取噪声与同义词合并难；必须设计人工纠错入口或白名单

### 方案 6：主题模型地图（Topic Modeling）
**核心思路**：学习“主题分布”，把文章放进主题空间，呈现主题之间距离与演化。
- 输入信号：标题/摘要/正文词袋（可清洗）
- 建图逻辑：LDA/NMF 或更轻量的聚类 -> 主题-文章关联；主题间按相似度连边
- 产出物：主题节点 + 文章挂载；可展示“主题热度随时间变化”
- 资源画像：中；离线可做；线上静态展示
- 风险/代价：主题可解释性需要人工命名；短文本效果可能一般

### 方案 7：时间演化图（Chronological Graph）
**核心思路**：强调“知识随时间增长的路径”，把文章按时间与依赖关系组织成学习路线。
- 输入信号：发布时间、系列、引用、关键概念出现时间
- 建图逻辑：时间轴为主；叠加引用/相似度形成跨时间边；可抽取“里程碑节点”
- 产出物：时间线地图（timeline graph）+ 每个阶段的关键文章
- 资源画像：低到中；依赖的“里程碑识别”可规则或模型
- 风险/代价：对写作节奏敏感；需要定义“阶段/里程碑”的标准

### 方案 8：用户行为图（点击/停留/路径）
**核心思路**：用真实读者行为挖掘“阅读路径”，把地图做成可提升转化的导航。
- 输入信号：访问日志/统计（页面路径、referrer、session 路径、停留时间）
- 建图逻辑：会话路径 -> 页面转移概率图；边权为转移次数/概率
- 产出物：阅读路径图 + “从 A 最常去 B” + 漏斗/入口出口
- 资源画像：中（需要统计与会话化处理）；但非常贴合“数据驱动迭代”
- 风险/代价：需要隐私合规与采集最小化；低流量时信号稀疏

### 方案 9：问答意图图（Query/Question Graph）
**核心思路**：把文章映射为“回答了哪些问题”，地图节点是问题/意图而不是文章。
- 输入信号：文章标题/小标题；自动生成 Q&A（规则或模型）；站内搜索词（若有）
- 建图逻辑：问题聚类（同义合并）-> 问题之间按依赖/先修关系连边
- 产出物：问题图（学习路径）+ 每个问题的文章集合
- 资源画像：中；离线生成；可做到高可用导出
- 风险/代价：问题生成与同义合并质量要求高；需要人工编辑入口提升稳定性

### 方案 10：二部图与投影（文章-概念/文章-代码/文章-引用资源）
**核心思路**：先构建二部图（文章-概念），再按不同投影生成不同地图（概念图/文章图）。
- 输入信号：标签、实体、代码符号、外链域名、引用书单等任意“可枚举项”
- 建图逻辑：构建二部图 -> 对文章侧或概念侧做投影（共享邻居越多边越强）
- 产出物：可切换的多地图（概念地图/文章地图/资源地图）
- 资源画像：低到中；结构通用、可扩展
- 风险/代价：需要定义“概念来源”与去噪规则，否则图会过密

### 方案 11：分层概念树（Ontology/Taxonomy）
**核心思路**：先有一个“分类体系”（树/有向无环图），文章挂载到叶子，并显示跨类连接。
- 输入信号：手工 taxonomy（YAML/JSON）；文章的分类标注；可用模型建议分类
- 建图逻辑：分类树为骨架；跨类边来自引用/相似度/共现
- 产出物：可折叠的知识树 + 交叉引用边
- 资源画像：低；长期稳定；适合 SEO（清晰的栏目与面包屑）
- 风险/代价：初始 taxonomy 设计需要一轮思考；后期需要少量治理

## 默认推荐的最小实现路径（资源友好）
- 第一步（零模型）：方案 2（内链图）+ 方案 3（标签共现）+ 基础可视化
- 第二步（引入 AI 但可控）：在构建期加方案 4（embedding kNN，优先只用标题+摘要）
- 第三步（需要更“知识图谱”）：按需引入方案 5（实体-关系抽取），并配套纠错/白名单
