# 站点 + 搜索 + Chat（向量库）+ 知识图谱（地图）一体化实现 Spec

## Why
网站内容越来越多后，需要一套“可发现系统”：既能像传统搜索一样可控、可解释，又能用 AI 补强语义检索与对话式问答，并用知识图谱/地图提供全局框架与探索式阅读路径，降低读者理解门槛并支持数据驱动迭代。

## What Changes
- 以“Markdown 原文为单一事实来源”为原则，建立统一的数据管线：内容 -> 元数据/索引 -> 搜索/Chat -> 知识地图
- 提供搜索能力（关键词检索 + 语义检索混合）与 Chat（RAG，带引用）
- 自动生成整体知识地图（可视化、可点击预览、可导出）
- 生成并维护：标签/关键词/摘要/标题改写建议（可选）/阅读顺序建议（学习路径）
- 记录可控的用户反馈信号（搜索/Chat 查询、点击/停留等），用于内容优化

## Impact
- Affected specs: 构建与发布、内容组织与发现、隐私与安全、运维与备份、分析闭环
- Affected code: SSG 构建脚本、索引生成器、轻量 API 服务、前端搜索/Chat/知识地图页面

## Design Principles
- 单一事实来源：文章原文仍以 md 文件存储；所有索引/图谱/摘要均可重建
- 低资源优先：2C/2G/3Mbps/40G 环境线上尽量“静态为主 + 极轻服务为辅”
- 可解释与可追溯：所有 AI 生成内容必须可回溯到证据（文章/段落引用、生成时间、模型版本标识）
- 可渐进演进：先做零模型 MVP（内链图 + 标签图 + 关键词检索），再引入向量库与 Chat
- 自建优先：不依赖必须付费的外部托管；AI API 由用户自行提供（不纳入成本）

## System Overview（推荐架构）

### 数据与产物（Artifacts）
- 内容源：`content/**/*.md`（含 frontmatter：title/date/tags/summary/series 等）
- 媒体：`static/media/**`（图床目录，URL 稳定）
- 统一数据模型规范：见 `./content-model.md`（frontmatter / derived fields / chunk / evidence）
- 索引产物（可重建）：
  - `search_lexical.db`：关键词检索索引（SQLite FTS5 或等价方案）
  - `vectors/`：向量库（推荐嵌入式文件型，如 LanceDB；Chroma 作为可选）
  - `graph/*.json`：知识地图（节点/边/权重/预览片段/证据）
  - `ai_cache/*.json`：AI 生成缓存（标签/摘要/问题列表/学习路径），带版本与时间戳
- 反馈数据（可选，可清理）：`feedback.db`（SQLite；存查询与点击的聚合/最小化数据）

### 运行形态
- 静态站点：Nginx/Caddy 托管 SSG 产物（强缓存 + 压缩 + 图片优化）
- 轻量 API（可选但推荐用于 Chat 与向量检索）：
  - `POST /api/search`：混合检索（关键词 + 向量 + rerank）
  - `POST /api/chat`：RAG 对话（返回答案 + 引用片段 + 推荐阅读顺序）
  - `GET /api/graph`：按需提供图谱子图（也可直接静态 JSON）

## ADDED Requirements

### Requirement: 混合搜索
系统 SHALL 提供混合搜索：关键词检索与语义检索同时可用，并返回带权重的合并结果。

#### Scenario: 搜索命中
- **WHEN** 用户输入查询词
- **THEN** 返回 TopN 文章结果，包含标题、摘要、匹配原因（关键词命中字段/语义相似度）与跳转链接

### Requirement: Chat（RAG + 引用）
系统 SHALL 提供 Chat 模式：在回答中引用具体文章与片段，并可一键跳转。

#### Scenario: Chat 提问
- **WHEN** 用户在 Chat 输入问题
- **THEN** 系统返回答案、引用列表（文章 + 片段）、以及“建议阅读顺序”（可为空）

### Requirement: 自动元数据生成
系统 SHALL 支持 AI agent 为文章生成标签与元数据（至少：tags、summary、keywords），并写入可重建的缓存产物，而不是改写原文作为唯一存储。

### Requirement: 整体知识地图（可视化 + 预览）
系统 SHALL 提供整体知识地图页面，支持按关键词/概念浏览节点，点击节点展示关联边与文章预览，并可跳转到文章。

### Requirement: 临时学习路径与临时子图
系统 SHALL 支持在 Chat 中为“我不熟悉这个领域”这类请求生成：
- 阅读顺序建议（文章列表 + 理由）
- 可选的临时子图（仅围绕本次问题的若干关键节点/文章）

### Requirement: 数据与反馈闭环
系统 SHOULD 记录最小化的用户反馈信号（搜索查询、点击的文章、Chat 查询与引用点击），并可导出用于内容优化。

### Requirement: 可运维与可恢复
系统 SHALL 明确备份策略：内容源、媒体、索引产物、反馈数据均可备份；索引与图谱可从内容源重建。

## 关键模块的实现取舍（建议）

### SSG（网站构建）
- 推荐：静态站点生成器（Hugo/Zola 等）产出纯静态 HTML，SEO 与性能最佳；动态后端仅负责搜索/Chat/图谱 API

### 关键词检索（“类似 Elasticsearch 的基础能力”）
- 推荐：SQLite FTS5（文件型、零运维、资源友好）作为 lexical index
- 目标：支持标题/正文/标签字段检索、BM25 排序、片段高亮

### 向量库（语义检索）
- 推荐：LanceDB（嵌入式文件型，便于自建与备份），把 embedding 向量与文章 chunk 的元信息一起存储
- 可选：Chroma（如果你更偏好其生态，但需要考虑常驻服务与资源）

### Chat 服务（RAG）
- 推荐：单个轻量 API 服务承担：
  - 查询 -> 混合检索 -> 组装上下文（带片段）-> 调用用户提供的 AI API -> 返回答案 + 引用 + 学习路径
- 约束：必须限制并发与请求大小；必须提供 API key 或鉴权；必须做速率限制

### 知识图谱（地图）
- 推荐“多信号并存”的地图体系：
  - 基础：内链/引用图（可解释）+ 标签共现图（概念视角）
  - 增强：embedding 近邻图（语义补强）
  - 进阶：问题图/实体图（需要纠错机制）
- 前端可视化：D3.js 或等价方案输出可交互图；图数据以 JSON 导出

## 实施阶段（从整体统筹的路线）

### 阶段 1：零模型 MVP（先把结构与闭环跑通）
- 站点：SSG + 图床目录 + SEO 基线
- 搜索：SQLite FTS5 索引 + 前端搜索页
- 知识地图：内链图 + 标签共现图 + D3 可视化 + 节点预览
- 反馈：仅记录聚合（例如按天的查询 Top、点击 Top）

### 阶段 2：向量检索与 Chat（把 AI 补强接上）
- agent：生成 tags/summary/keywords；生成 embedding（优先标题+摘要，再到正文 chunk）
- 混合检索：lexical + vector 合并；支持“为什么命中”的解释字段
- Chat：RAG + 引用；“不熟悉”触发学习路径生成

### 阶段 3：整体知识体系增强（更像 Obsidian 图谱）
- 引入问题图或实体图（二选一先做）并提供纠错入口
- 支持临时子图（按 query 或选中文章集合生成），并可选择保存为“公共导览图”
