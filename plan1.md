### 考虑方面 1：个人网站

**目标：** 搭建一个以**每日文章发布**为核心的个人博客网站。

**核心约束与需求：**

1. **极简部署与运维：** 优先考虑部署简便性、**极低维护成本**和长期稳定性（希望“省心”）。
2. **严格资源限制：** 目标部署环境为现有**低配云服务器 (2核 CPU / 2GB RAM / 3Mbps 带宽 / 40GB 磁盘)**，所有组件必须高效利用有限资源（CPU、内存、磁盘、网络）。
3. **关键功能需求：**
   - 高性能响应（尤其在资源受限环境下）。
   - **内置访问量统计**（需提供详细数据用于分析）。
   - **必需图床功能**（用于存储和引用文章中的图片）。
   - **强 SEO 友好性**（利于搜索引擎收录和排名）。
4. **数据驱动迭代：** 基于访问量等统计信息进行**内容优化**（即“小步快跑”式迭代）。
5. **技术栈排除：** **不考虑 PHP、Java 及其重型生态**。
6. **部署模式偏好：** **首选自建部署**于现有服务器；托管方案（如 Vercel, Netlify）仅作为备选了解，**非主要方向**。
7. **成本控制：** **预算敏感**，力求零成本或极低成本（尤其避免额外付费服务）。
8. **现状：** 已有符合上述配置的云服务器（无域名）。

**核心问题：**
在满足以上**所有约束条件**的前提下，**推荐最合适的建站方法和技术栈**。

### 考虑方面2：功能扩展

背景：个人网站，分享知识，而用好 ai 的程度与自身水平有关，因为有知道知识框架，知道什么是好的坏的(判断力或者说 taste)，我的个人网站能降低这个门槛，让没那么了解我掌握内容的人也能按图索骥地去增长和提高自己。

具体做法：搜索功能(ai chat 模式)，以及整体框架一览(类似 obsidian 知识图谱)，有一个关键词的图(边和节点)

- 除了提供基础的 类似 elasticsearch 的搜索功能，还用 ai 补强，能根据用户的需求能查到，用向量数据库（lancedb/chromadb）之类的应该可以；
- 让 ai agent 给文章生成标签和元数据，将一些设计的字段放到向量数据库，原文还是以 md 文件存储，知识图谱也是自动生成的（考虑 D3.js）；
- 对我的以往内容不熟悉的可以根据我的知识图谱的脉络，能够知道有什么关键词，可以看到这些关键词对应知识体系的框架是怎样的，看到我提供的知识图谱，知道之后应该怎么问，以及探索性的点击对应的节点进去看看内容(最好提供预览功能)；
- 进一步补强的功能，假设用户在 ai chat 中，直接说自己不熟，那么提供几篇文章的阅读顺序(如果能临时生成一条临时的可视的知识图谱)，以及可以考虑直接根据几篇内容生成只能聚合摘要(同时也会保存在服务器中，作为用户搜索的统计和反馈)，参考 ima 的各种功能
- ai API 我会自己另外提供，这里不纳入成本考虑
