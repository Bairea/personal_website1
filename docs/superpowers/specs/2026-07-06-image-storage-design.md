# 图片存储策略设计

**日期**：2026-07-06
**状态**：已确认
**范围**：决定网站图片的长期存储架构，评估是否需要图床

## 背景

网站目前图片极少（2 个 SVG），存储在 `static/media/`，Hugo 原样复制到 `public/`，Express 直接提供静态文件。需要评估：当文章量从几十篇增长到上千上万篇时，图片存储架构是否需要调整，是否有必要引入图床或对象存储。

## 核心结论

**不需要图床。** 图片继续放 `static/media/`，纳入 Git 管理，是本场景下的最优解。

理由：
1. 技术配图天然体量小（平均 ~150KB/张），万篇文章仅占 ~3GB
2. 个人站流量低，带宽不是瓶颈
3. 图床引入第三方依赖，10 年维度上服务可能关停/改价/限制
4. Git 管理图片提供版本控制和简单备份，部署只需 `git clone`

## 方案对比

### 方案 A：图片放 `static/media/`，Git 管理（选定 ✅）

- 零依赖，零复杂度
- Git 版本控制，误删可恢复
- 备份简单：`content/` + `static/media/` 一套 `rsync` 搞定
- 迁移容易：`git clone` + `npm run build` 即可
- 3GB/万篇的体量在 Git 舒适范围内

### 方案 B：图片放本地但排除出 Git

- 解决了一个不存在的问题（Git 仓库太大），引入了真实的运维复杂度
- 需要两套备份策略，容易遗漏
- 图片变更不可追溯

### 方案 C：图床/对象存储

- 10 年风险：第三方服务可能关停/改价
- 与自托管偏好矛盾
- 迁移成本高，需要批量改 URL
- 解决的是大图片+高带宽问题，本场景不存在

## 设计详情

### 1. 图片存储与组织

**存储位置**：`static/media/`（不变）

**目录结构**：按 `/<yyyy>/<mm>/<slug>.<ext>` 组织：

```
static/media/
├── 2026/
│   ├── 03/
│   │   └── discovery-pipeline.webp
│   ├── 07/
│   │   ├── hugo-quickstart-roadmap.svg
│   │   └── map-design-entity.svg
│   └── ...
├── 2027/
│   └── ...
└── demo/          ← 现有演示图片，保留
    ├── roadmap.svg
    └── entity-map.svg
```

**命名规则**：
- 文件名 = `<文章slug>-<描述>.<ext>`
- 避免纯数字或无意义名称
- 同一文章多张图用不同描述后缀区分

**格式偏好**（按优先级）：
1. SVG — 架构图、流程图、示意图
2. WebP — 截图、光栅图
3. PNG — 必须透明背景时
4. JPG — 照片（罕见场景）

**引用方式**：Markdown 中用绝对路径：
```md
![示例图](/media/2026/07/hugo-quickstart-roadmap.svg)
```

**Git 管理**：图片纳入 Git 版本控制，不做 `.gitignore` 排除，不使用 Git LFS。

### 2. 图片优化策略

**现阶段（0-500 篇）**：不做优化。技术配图天然就小，优化步骤增加构建复杂度不值得。

**后续可加的优化（500+ 篇时评估）**：

| 优化 | 效果 | 实现方式 |
|---|---|---|
| SVG 优化 | 减小 20-50% | `svgo` 集成到 `npm run build` |
| PNG → WebP 转换 | 减小 50-70% | `sharp` 集成到构建流程 |
| 图片压缩 | PNG 减小 30-50% | `sharp` 或 `imagemin` |

**实现方式**：在 `scripts/` 下新增 `optimize-images.js`，作为 `npm run build:images` 步骤，在 Hugo 构建前处理 `static/media/` 中的图片。优化后的文件覆盖源文件（Git 保留历史版本）。优化操作幂等，不累积质量损失。

### 3. 长期容量规划

| 文章量 | 图片数（估） | 存储占用 | Git 仓库大小 | VPS 磁盘占比（40GB） |
|---|---|---|---|---|
| 100 篇 | ~200 | ~30MB | ~35MB | <0.1% |
| 1,000 篇 | ~2,000 | ~300MB | ~350MB | <1% |
| 5,000 篇 | ~10,000 | ~1.5GB | ~1.8GB | ~4.5% |
| 10,000 篇 | ~20,000 | ~3GB | ~3.5GB | ~9% |

即使到万篇级别，图片也只占磁盘 ~9%，完全不是瓶颈。

### 4. 运维考量

**Git 仓库大小**：3.5GB 在 Git 正常范围内（Linux 内核 ~4GB+）。如 `git clone` 慢可用 `--depth 1` 浅克隆。

**备份**：现有策略（备份 `content/` + `static/media/`）已覆盖，无需额外步骤。

**带宽**：个人站日均几百访问，每页图片 <500KB，日均带宽 <200MB，VPS 带宽够用。如未来流量暴增，优先考虑 Cloudflare 免费 CDN（加缓存层，不改存储架构）。

**迁移**：`git clone` + `npm run build` 即可。图片用相对路径 `/media/...`，不受域名影响。
